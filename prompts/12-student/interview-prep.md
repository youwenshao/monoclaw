# InterviewPrep

## Tool Name & Overview

InterviewPrep is a local coding practice and interview preparation tool that presents LeetCode-style algorithmic problems, provides progressive hints, generates detailed explanations of solutions, and tracks the student's performance to identify weak areas. It uses the local LLM for personalized explanations and adaptive difficulty, ensuring students can practice coding interviews without an internet connection or subscription fees.

## Target User

Computer science students, coding bootcamp graduates, and junior developers in Hong Kong preparing for technical interviews at tech companies, banks (HSBC, Standard Chartered tech divisions), and startups, who want structured practice with AI-powered guidance.

## Core Features

- **Problem Library**: Curated library of 200+ algorithm and data structure problems organized by topic (arrays, strings, trees, graphs, dynamic programming, etc.) and difficulty (easy, medium, hard)
- **Progressive Hints**: When stuck, students can request hints that gradually reveal the approach — from high-level strategy to specific algorithm choice to implementation guidance
- **Solution Explanation**: After solving (or giving up), receive a detailed walkthrough of the optimal solution with time/space complexity analysis, generated by the local LLM
- **Code Execution**: Runs student solutions against test cases locally using Python/JavaScript subprocess; shows pass/fail results with edge case testing
- **Weak Area Tracking**: Analyzes solve rate, time spent, and hint usage per topic to identify areas needing more practice; generates personalized practice plans
- **Mock Interview Mode**: Timed 45-minute sessions simulating real interview conditions with 2 problems, no hints, and a post-session performance review

## Tech Stack

- **LLM**: MLX local inference (Qwen-2.5-Coder-7B) for hint generation, solution explanation, and code review
- **Code Execution**: subprocess with timeout and memory limits for running Python/JavaScript solutions safely
- **Database**: SQLite for problem library, student progress, and performance analytics
- **UI**: Streamlit with code editor (streamlit-ace), problem display, and progress dashboard
- **Testing**: Custom test case runner with input/output comparison and edge case generation

## Key Integrations

- **Local LLM (MLX)**: Hint generation, solution explanations, and code review — all on-device
- **Code Execution**: Local Python/JavaScript subprocess for running and testing student solutions
- **No external dependencies**: Works fully offline — no LeetCode subscription needed

## HK-Specific Requirements

- HK tech interview landscape: Students in HK interview at international tech companies (Google, Meta, Amazon HK offices), local tech companies (Lalamove, GoGoX, Klook), banks (HSBC Technology, Standard Chartered), and fintech firms — problem difficulty should cover this range
- Interview language: Technical interviews in HK are conducted in English; the tool should present problems and accept solutions in English, but allow LLM explanations in Traditional Chinese if requested
- Common HK interview formats: Many HK companies use HackerRank or Codility for online assessments before in-person interviews — the mock interview mode should simulate this format
- University course alignment: Problem topics align with common HK CS curriculum (COMP courses at HKUST/HKU/CUHK cover algorithms, data structures, and discrete math)

## Data Model

```sql
CREATE TABLE problems (
    id INTEGER PRIMARY KEY,
    title TEXT NOT NULL,
    slug TEXT UNIQUE,
    description TEXT,
    difficulty TEXT CHECK(difficulty IN ('easy','medium','hard')),
    topic TEXT,
    subtopic TEXT,
    example_input TEXT,
    example_output TEXT,
    constraints TEXT,
    optimal_time_complexity TEXT,
    optimal_space_complexity TEXT,
    solution_python TEXT,
    solution_javascript TEXT,
    hints TEXT,  -- JSON array of progressive hints
    test_cases TEXT  -- JSON array of {input, expected_output}
);

CREATE TABLE attempts (
    id INTEGER PRIMARY KEY,
    problem_id INTEGER REFERENCES problems(id),
    code TEXT,
    language TEXT CHECK(language IN ('python','javascript')),
    passed_tests INTEGER,
    total_tests INTEGER,
    is_correct BOOLEAN,
    hints_used INTEGER DEFAULT 0,
    time_spent_seconds INTEGER,
    submitted_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE progress (
    id INTEGER PRIMARY KEY,
    topic TEXT,
    problems_attempted INTEGER DEFAULT 0,
    problems_solved INTEGER DEFAULT 0,
    avg_time_seconds REAL,
    avg_hints_used REAL,
    solve_rate REAL,
    last_practiced TIMESTAMP,
    strength_level TEXT CHECK(strength_level IN ('weak','developing','strong')) DEFAULT 'weak'
);

CREATE TABLE mock_interviews (
    id INTEGER PRIMARY KEY,
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    problems TEXT,  -- JSON array of problem IDs
    results TEXT,   -- JSON: per-problem pass/fail and time
    overall_score REAL,
    feedback TEXT
);

CREATE TABLE study_plans (
    id INTEGER PRIMARY KEY,
    generated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    focus_topics TEXT,  -- JSON array
    daily_problems INTEGER DEFAULT 3,
    duration_days INTEGER DEFAULT 14,
    plan_details TEXT  -- JSON: daily problem assignments
);
```

## Testing Criteria

- [ ] Problem display renders markdown description, examples, and constraints correctly in Streamlit
- [ ] Code runner executes a correct Python two-sum solution and reports all test cases passed
- [ ] Code runner catches an incorrect solution (wrong output) and reports which test cases failed
- [ ] Hint engine provides 3 progressive hints — each more specific than the last
- [ ] Solution explainer generates a clear walkthrough with complexity analysis for a medium-difficulty problem
- [ ] Weakness analyzer correctly identifies "dynamic programming" as a weak area when solve rate is below 30%
- [ ] Mock interview mode enforces 45-minute time limit and prevents hint access

## Implementation Notes

- Code execution safety: run student code in a subprocess with `ulimit` constraints — 5-second timeout, 256MB memory limit; never use `eval()` or `exec()` in the main process
- Problem library: start with 200 problems covering the most common interview topics; store as JSON files organized by topic for easy expansion
- Progressive hints: structure as 3 levels — Level 1: "Think about what data structure would help" → Level 2: "Consider using a hash map to store..." → Level 3: "Iterate through the array, for each element check if target - element exists in the map"
- LLM for explanations: after the student solves or views the solution, generate a personalized explanation that addresses their specific approach (if they submitted code) or walks through the optimal approach
- Spaced repetition for weak areas: resurface problems from weak topics more frequently in practice recommendations
- Memory budget: ~5GB (Qwen-2.5-Coder-7B for explanations + Streamlit + code execution overhead)
- Mock interview scoring: weight correctness (60%), time efficiency (20%), and code quality (20%) — use LLM to assess code quality (naming, structure, edge case handling)
- Consider adding a "system design" question bank for more senior interview preparation, using the LLM for interactive discussion-based practice
